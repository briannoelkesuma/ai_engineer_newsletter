Build Agents That Run for Hours (Without Losing the Plot) — Ash Prabaker & Andrew Wilson, Anthropic

20260518

High-Level Summary

This presentation by Ash Prabaker and Andrew Wilson from Anthropic discusses the challenges and solutions for building long-running agents that can operate for hours without losing coherence. They cover the evolution of Claude models and the Agent SDK, highlighting key improvements in context management, planning, and self-evaluation. The speakers introduce harness patterns, including the generator-evaluator contract, and emphasize the importance of reading traces for debugging and adjusting harnesses as models evolve. The presentation also touches on practical tips for building agent harnesses and key takeaways for achieving long-running agent success.

Detailed Learnings

0:00 Introduction and speakers - Ash Prabaker and Andrew Wilson from Anthropic's applied AI team introduce the topic of building long-running agents, inspired by their recent blog post. They aim to share details about the harnesses used to create agents capable of running for extended periods (5-6 hours or more). 1:21 Overview of long-running agents - Andrew will discuss the history and primitives that have enabled agents to run for extended periods. Ash will then cover more experimental harness patterns and examples. 2:29 Challenges: Context, Planning, and Judgment - Andrew presents a quote from Boris, the creator of Claude Code, highlighting the significant progress made in a year, from struggling with bash commands to writing its own code and running for days. - He frames the problem of building long-running agents around three main challenges: context, planning, and judgment. - Context: Finite context windows lead to amnesia in new sessions and context rot as sessions progress. Models may also exhibit context sense anxiety near the end of their context window. - Planning: Models often struggle with planning, attempting to do everything at once or leaving tasks half-finished due to context limitations. - Judgment: Models are poor at judging their own output, often overestimating the completeness or correctness of their work. 4:14 Two approaches: Model updates vs. Harness evolution - Two approaches to address these challenges are discussed: improving the model itself and evolving the harness (scaffolding around the model). - A meter chart illustrates the progress in agent runtime with minimal scaffolding, showing an increase from 1 hour with Opus 3.7 to 12 hours with Opus 4.6. - The Agent SDK provides primitives for building harnesses, including the core agent loop, tool integration, sub-agent delegation, context management, and a permission system. - Harnesses evolve as models improve, requiring continuous adaptation and refinement. 5:58 Prehistory: Sonnet 3.5, Computer Use, and MCP - Before Claude Code, Sonnet 3.5 showed promise in coding by verifying and iterating on its work. Computer use allowed agents to interact with interfaces, take screenshots, and test code. - MCP (Model Control Plane) spec enabled agents to use tools. 6:34 The evolution of Claude Code - Claude Code was released in research preview with Sonnet 3.7, aiming to understand how developers use Claude for coding and inform future model improvements. - The goal was experimental, with the intention of informing improvements to the base model. - The slides highlight the focus of each release (context, planning, verification) and relevant statistics. 7:55 The Ralph loop technique - The Ralph Wiggum loop technique, popularized by Jeffrey Huntley, involves running a prompt in a loop until tasks are complete. It gained traction later in the year. - The technique involves breaking down a prompt into features, selecting a task, and starting a new session with a fresh context window. - Anthropic created its own Ralph Loop plugin within the Claude Code harness, running within a single Claude Code session and relying on compaction instead of fresh context windows. - The plugin sets max iterations and a safe word, with a stop hook intercepting Claude's typical stopping point and continuing until exit criteria are met. 9:49 Sonnet 4.5, Agent SDK, and checkpoints - Sonnet 4.5 improved context awareness, tracking token consumption and managing context as it approached the end of the window. - Claude Code 2.0 introduced checkpoints for tracking code over time and rewinding to previous states. - The Claude Code SDK was renamed to the Agent SDK to reflect its broader applicability beyond coding. - Claude Sonnet 4.5 could run for about 30 hours. 10:49 Opus 4.5 and the role of sub-agents - Haiku 4.5 and Opus 4.5 made running multiple sub-agents economical, with Opus 4.5 excelling at planning. - Opus 4.5 could be used for planning, while Sonnet 4.5 served as the workhorse for executing code. - Skills were introduced, effectively using context with progressive disclosure, loading only the front matter initially and the rest upon instantiation. - Programmatic tool calling improved context usage by writing code on the fly to run tool calls and retrieve final results. 12:05 First long-running agent patterns - The first blog post on long-running agents was released, detailing how to build them. - A human would provide a vague prompt (e.g., "write me a browser"), and an initializer agent would break it down into persistent artifacts: - A feature list (featurelist.json) to avoid overwriting. - A progress file. - A Git repository. - An init script. - A flag for feature completion. - The harness loop involves: - Getting bearings in a fresh context window. - Running a smoke test or init script. - Picking one unfinished feature. - Implementing the feature. - Testing (e.g., using Puppeteer). - Writing a Git commit and updating the feature state if tests pass. - The loop continues with fresh context windows, persistent artifacts, verification loops, and upfront planning. 14:20 Opus 4.6, Agent Teams, and server-side compaction - Opus 4.6 and Sonnet 4.6 were released, with Sonnet 4.6 offering Opus-level intelligence at a lower price. - Opus 4.6 was highly agentic, excelling at planning and running for longer durations (12 hours with a simple harness). - Agent Teams were introduced as a general-purpose way to scaffold custom agents, allowing sub-agents to communicate with each other and report back to the main agent only when required. - Server-side compaction enabled models to run indefinitely, with compaction happening on the server side. - The context window was expanded to 1 million tokens. 17:28 State-of-the-art harness patterns - Ash discusses harness patterns used internally to build one-shot demo apps and experiment with autonomous work in post-training and RL. - The general idea is inspired by GANs (Generative Adversarial Networks), using a generator and a discriminator with adversarial pressure. - The generator builds, the evaluator grades, and the context windows, system prompts, and jobs are split entirely. - The evaluator uses Playwright to open live pages, click around, and provide critique back to the generator in a loop. - This contrasts with using a single Claude Code session and telling it to check its own work. 21:30 Evaluating subjective output with rubrics - Tuning a standalone critic to be harsh is more tractable than tuning a builder to be self-critical. - The analogy is drawn to humans, where critiquing artwork or meals is easier than creating them. - Designing critics involves creating a rubric with criteria like design, originality, craft, and functionality. - Weights are assigned to these criteria, with a focus on design and originality to prevent AI slop aesthetics. - Calibration is done with few-shot examples on reference sites to align the evaluator's taste with the desired aesthetic. 23:44 Introducing the 'Planner' role - An example is shown of a model going through the generator-evaluator loop, launching Playwright, navigating, taking screenshots, scoring on the criteria, writing critique, and handing back to the generator. - The examples are HTML and CSS only, iterated over 4-5 hours and 5-15 rounds. - The GAN-style harness can pivot and try again from scratch if the generator gets stuck on a criterion, unlike single-pass generation or RALF loops that keep patching the same thing. - To go from nice pages to fully working apps, a planner role is added. - The planner takes a one-line prompt and breaks it down into a high-level spec, specifying the general workflow into a series of sprints. - It avoids planning granular technical details to prevent cascading errors. 25:04 The generator-evaluator contract - The setup resembles a PM, IC, and QA org structure, with each role having its own context window. - Before building, the generator and evaluator negotiate what "done" means. - The generator proposes a feature and verification tests, and the evaluator pushes back if the scope is too big or the tests are too weak. - This back-and-forth occurs via files on disk until both agree. - The evaluator grades against the agreed-upon contract, not the original spec. - This bridges user stories and tangible, testable assertions without over-specifying upfront. - This is a key innovation missing from the Ralph loop, which had a fixed plan without adversarial argument. 31:28 Specificity in contracts and debugging traces - A comparison is made between a solo loop and the harness with the prompt "build a retro game maker." - The difference is attributed to scaffolding. - The evaluator catches basic issues like fast API route ordering, Boolean logic bugs, and problems with the delete key, which are only caught because the evaluator is using the app. - Significant time is spent tuning prompts to address layout bugs and edge cases. - The primary debugging loop involves reading traces to identify divergences between the model's judgment and human judgment, then tuning the prompt accordingly. - Agent transcripts are piped into files, graphed with another agent, or played through by another agent to update prompts. - Adjusting the harness as models improve involves understanding the spiky behaviors of each model and filling the gaps. - Context resetting between sessions was dropped as Opus 4.6 doesn't exhibit context anxiety. - Sprint decomposition, critical for Opus 4.5, was less necessary for Opus 4.6. - The evaluator cadence was reduced to running at the end of a one-shot generation. 37:56 How to build your own agent harness - The final setup involves the planner-generator-evaluator loop, with a simplified set of components. - A file system is used for shared state instead of relying on context windows. - An example is shown of a DAW (digital audio workstation) app built with the simplified harness, demonstrating the agent setting tempo, key, melody, and drum tracks. - The evaluator tests the app, and the music is trash, but the app is good and fleshed out. - To build your own agent harness, you can use auto mode, custom sub-agents, Playwright MTP or Claude for Chrome MTP, and skills. 39:01 Key takeaways for long-running agents - Key takeaways: - Self-evaluation is a trap; use an adversarial evaluator. - Compaction doesn't equal coherence; structured hand-offs and clean contexts are better. - Subjective quality is gradable; write down your strong views. - Why Your AI UX Is Broken (and It's Not the Model's Fault) — Mike Christensen, Ably

20260517

High-Level Summary

Mike Christensen from Ably discusses the limitations of direct HTTP streaming (specifically Server-Sent Events - SSE) in AI chat applications and proposes a more robust solution using durable sessions. The core problem lies in the tight coupling between the client connection and the agent's response stream, leading to issues with resumability, multi-device support, and concurrent agent activity. The solution involves decoupling the agent and client layers with a shared, persistent 'durable session' resource, enabling resilient delivery, continuity across devices, and live control. He demonstrates these concepts with a demo showcasing multi-tab sync, network disconnect recovery, concurrent agents, and human agent handoff.

Detailed Learnings

The video addresses the challenges of building robust AI chat applications, focusing on the limitations of direct HTTP streaming and proposing a durable sessions approach. - **[00:00:00] Introduction:** The speaker introduces the common pattern of AI chat applications where a client sends a message to an agent, and the agent streams a response back. He highlights that the default implementation often relies on direct HTTP streaming using Server-Sent Events (SSE). - **[00:00:24] Direct HTTP Streaming (SSE):** The client establishes a persistent, point-to-point connection with the agent. The agent invokes the LLM, obtains a stream of events, and pipes these events back to the client over the connection. While easy to implement initially, this approach is fundamentally oriented towards a single client, single connection, single agent paradigm. - **[00:00:50] Limitations of the Single Connection Model:** The speaker argues that the single-connection model limits the quality and richness of AI experiences. He introduces himself as Mike Christensen from Ably, a real-time messaging platform, and mentions their experience working with numerous companies building AI products. - **[00:01:24] Key Capabilities for Great AI Experiences:** Based on their observations, the speaker identifies three foundational capabilities that distinguish a fragile demo from a great AI product experience: resilient delivery (surviving disconnections), continuity across surfaces (multi-device sync), and live control (interacting with the agent while it's working). - **[00:01:48] Resilient Delivery:** This refers to the ability of streams to survive disconnections. For example, when a user's mobile device switches from Wi-Fi to 5G, the connection is severed. The goal is for clients to reconnect and resume from where they left off. - **[00:02:03] Continuity Across Surfaces:** The conversation session should seamlessly follow the user across different devices and tabs. The session should be fully in sync, including any live activity, regardless of the device being used. - **[00:02:17] Live Control:** The best AI products go beyond a simple request-response pattern. They allow users to communicate with the agent while it's working, enabling steering and follow-up requests. This requires visibility into what the agent is doing and the ability for clients to interact with the agent during the process. - **[00:02:35] Why Direct HTTP Streaming Fails:** The root cause of the limitations is that everything is coupled to a single request. The health of the response stream is tied to the health of the client's connection. If the connection drops, the stream is lost. The connection becomes a private pipe, preventing visibility from other tabs or devices. Other clients cannot interact with the agent. - **[00:02:59] Durable Sessions:** The speaker introduces the concept of durable sessions as a solution.
